Week 1 Practical Guide

Sprint 0 — “Startup Onboarding”

1 · One‑Page Project Charter

Every squad's measurable expectations.

How to fill each section ▾
  • Vision — one sentence that states why this product should exist.
  • Problem Statement — articulate pain points, not features.
  • Personas — give each target user a memorable name & KPI (The Investing Board in our Case).
  • Success Metrics (SMART) — Specific, Measurable, Achievable, Relevant, Time‑bound. Example: “Cut model training time by 60% before 31 Dec 2025”.
  • MVP Scope — list the minimum slice that demonstrates value. Aim for something you can demo in ≤ 11 weeks.
  • Risks & Mitigations — brainstorm what could derail you (data quality, regulatory, model bias) and write a concrete safety net.
# Project Charter — {Squad Name}
## Vision
*A single‑sentence north‑star.*

2 · Form Diverse Squads with rotating roles

Cross‑functional squads (of 5 students) speed up delivery because fewer hand‑offs are needed. Ensure each capability below is covered as well as passed to the next team member.

Skill DimensionExpectation
Team Leader= 1 who takes care of stuff and holds accountable.
Quant= 1 who handles probability & time‑series models.
DevOps/Data= 1 who owns Docker, CI/CD (Continuous Integration/Continuous Delivery), data pipelines.
Domain= 1 with Economics/Finance context.
Storyteller= 1 who can demo insights & create visuals.

3 · Initialise the GitHub Repository

    You are going to create a profile. Then, we are going to keep it simple at first by uploading any relevant docs as well as Jupyter Notebook (you are going to need Python first) files.

4 · Set Up the JIRA Board (each team)

JIRA keeps backlog, sprint cadence & metrics in one place. Follow the steps below (approx. 15 min).

StepAction
4.1Create your profile.
4.2Agree upon the backlog and the various items that need to be done.
4.3Create a list of tasks, related to the items.
4.4Create a sprint with the tasks you want to complete in the (first) week.
4.6Start Sprint # (# week). Add capacity (h) so burndown chart works.
4.6Note your capacity (h) for the sprint and commit to that.
4.7Add a burndown chart to the board.
4.8Attach any necessary docs and communicate via the chat.

5 · Squad‑wide Definition of Done

A shared Definition of Done guarantees transparency across what had to be done .

  • All code reviewed (≥ 1 approval) and PR merged via squash.
  • Unit tests pass in GitHub Actions.
  • Create a short progress report of what has been done and present it in 5 min.
